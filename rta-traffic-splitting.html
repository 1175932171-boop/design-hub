RTA策略验证体系

分流机制设计 · 桶内梯度校验 · 版本整体对照

适用范围：媒体侧固定10分桶约束（巨量、腾讯等渠道） 文档状态：评审稿 · 待算法/数据团队对齐 更新日期：2026年6月

一、背景与工程约束

RTA（Real-Time API）是广告主在媒体竞价过程中，通过实时接口针对每一次曝光请求返回差异化出价的机制。核心目标是：根据用户价值精准出价，花合理的价格买到合适的用户。

媒体约束
头条（巨量）、腾讯、百度、快手等渠道均提供 固定10个分桶，这是不可绕过的工程硬约束。一个RTA实验对应一个RTA策略，一个策略最多覆盖10个分桶，10个分桶对应最多10套差异化出价档位。
验证目标
验证两类问题：① 同一策略版本内，10档出价梯度是否合理（性价比最优区间在哪）；② 策略版本迭代后，新版本整体是否优于旧版本。
跨平台挑战
同一用户可能在头条、腾讯、百度、快手多个平台同时产生曝光请求，需保证该用户在所有平台归属同一实验桶，否则跨平台对照无效。

二、分流ID选择：userid vs deviceid

2.1 三类RTA请求场景

场景描述占比参考
类型① 有userid已登录用户，可通过ID Mapping将媒体deviceid映射到广告主内部userid~40–70%
类型② 仅有deviceid匿名用户，RTA请求只含IDFA/OAID，无法关联到内部userid~20–40%
类型③ 无任何IDiOS限制追踪或特殊设备，IDFA全零，无法获取任何有效标识~5–10%

2.2 两种分流方案的核心矛盾

评估维度userid 分流deviceid 分流
用户分层浓度均匀✓ 数学保证✗ 多设备导致失真
跨平台桶归属一致✓ 完全一致✗ 不同平台可能不同桶
用户信号完整性✓ 单用户信号聚合✗ 多设备信号分裂至多个桶
流量覆盖率仅限登录用户✓ 含匿名用户，覆盖更广
分析可解释性✓ 用户级归因清晰✗ 跨桶归因模糊
deviceid多设备分裂的具体危害：S级高价值用户若持有3台设备（d1/d2/d3），deviceid哈希后可能落入桶2、桶7、桶4三个不同桶。该用户的S级价值信号被稀释到3个桶，每个桶的S级"用户浓度"均被虚增或虚减，导致"初始浓度一致性"从根本上无法保证。
结论：从"各桶人群分层初始浓度绝对一致"的实验置信要求出发，userid分流是唯一能数学保证浓度均匀性的方案。deviceid的高覆盖率优势，是以浓度准确性和用户信号完整性为代价换来的，在需要初始浓度置信的前提下，此代价不可接受。

三、分层哈希设计（初始浓度一致性保障）

3.1 问题：纯哈希无法保证各层浓度均匀

即便使用userid，若直接执行 bucket = hash(userid) % 10，也只能保证各桶用户数量均匀，无法保证各桶内S/A/B/C各层的占比一致。例如S级高价值用户恰好集中在桶0–2，则任何桶间对比的效果差异都可能是"初始人群偏差"而非"出价策略"造成的。

3.2 解法：先分层，层内再分桶

分层哈希核心逻辑：对每个用户，先按价值模型归入tier，再在该tier内部独立执行哈希分桶。这样数学上保证：S层1000人 → 每桶约100人；A层3000人 → 每桶约300人。所有桶的人群分层浓度在期望上完全一致。 
Step 1：价值模型分层
    tier = value_model(userid)
    → {S: LTV Top 10%,  A: 10%–30%,  B: 30%–60%,  C: 60%以下}

Step 2：层内哈希分桶（各层独立执行）
    bucket_id = MurmurHash3(userid) % 10
    → 层内随机均匀分配，[0, 9] 各桶均等

结果：任意tier T，bucket_id ∈ [0,9] 中每个桶约含 T层总量 × 10%
    → 10个桶的 S/A/B/C 层占比在统计上完全一致

3.3 分层 vs 分桶的独立性（关键概念解耦）

维度分桶（Bucket）分层（Tier）
决定依据userid哈希，永久固定用户行为/价值模型，动态更新
含义该用户参与哪个实验组对该用户当前应出多少价
随时间变化不变（保证实验追溯性）随活跃度/消费行为变化
实验分析依据用实验启动时（T=0）的分桶快照用实验启动时的tier快照做起点校验
用户在实验运行期间从B层升至A层，其bucket不变（保证追溯性），而出价策略按新tier执行（保证精准性）。初始浓度一致性校验，应用T=0时刻的tier快照，而非实时tier。

3.4 均匀性验证（实验启动前 AA 测试）

验证项方法通过阈值
分桶用户数均匀性χ²检验（10桶 vs 期望10%）p > 0.05
各桶tier分布一致性KS检验，各层占比跨桶对比偏差 < 3%
历史行为特征均匀性各桶历史CVR/CTR方差检验变异系数 CV < 10%

五、10桶分配结构

5.1 标准实验分配（验证单一出价策略）

桶 0–3实验组40%新RTA出价策略，差异化出价干预

RTA增量ROI = （HoldOut组转化成本 − 实验组转化成本）/ HoldOut组成本

风险场景问题描述解法
用户跨设备同用户多设备被分到不同桶强制userid分桶，禁用deviceid作为主分流ID

七、核心澄清：浓度均匀的两阶段标准

"客群浓度需要均匀"这个要求，只在实验启动前（曝光前）成立。实验运行后（曝光后）浓度产生差异，恰恰是验证出价策略是否有效的核心证据，而不是需要消除的噪音。
阶段浓度应该如何原因
阶段1：分桶分配时（曝光前） 必须均匀 保证"起点公平"——不做任何出价干预时，10个桶里潜在高价值用户占比应一致，否则无法判断后续浓度差异是出价策略起作用，还是分桶本身有偏
阶段2：策略运行后（曝光后） 不能强求均匀，差异才是结果 RTA出价的作用机制是：高出价在媒体竞价中更容易竞得高价值用户曝光。若运行后浓度仍均匀，等价于否定了RTA出价策略本身的有效性
关键原则："控制变量"只能用来控制起点（分桶前的潜在客群分布），不能用来控制结果（曝光后的实际浓度）。后者是因变量，是用来做赛马判断的依据，强行把它控制均匀，等于把实验结果本身抹平了。

八、层面一：桶内出价梯度校验

8.1 适用场景

在单一RTA策略版本内部，验证10档出价梯度是否合理：出价越高，是否真的竞得了浓度越高的高价值用户；出价响应曲线是否符合预期形状。

8.2 起点校验（实验启动前必须执行）

1
按pLTV排序将用户分层（与正式实验使用同一套排序逻辑）
2
在切换差异化出价之前，统计10个桶各自的潜在高价值用户（b浓度）占比
3
校验10个桶占比差异是否在可接受阈值内（如 ±2%），运行χ²检验（p > 0.05）
4
通过 → 进入正式赛马阶段；不通过 → 排查分桶hash逻辑或pLTV排序偏差

8.3 正式赛马：出价响应曲线校验

不是简单比较"哪个桶浓度最高"，而是把b浓度对出价水平做曲线拟合，校验响应曲线的合理性：

校验项预期表现异常信号
单调性出价越高，b浓度越高（或至少不显著低于低出价桶）出现非单调（如3号桶浓度高于1号桶），提示出价梯度或pLTV排序有问题
边际递减浓度提升幅度随出价升高而递减，符合正常竞价市场规律边际效应不递减甚至递增，可能是出价档位间隔设计不合理
拐点定位找到"浓度提升 / 出价增量"性价比最优的区间用于后续调权放量参考

8.4 输出物

  • 各桶b浓度排序与出价响应曲线图
  • 性价比最优的出价区间，作为后续该桶流量放大的依据
  • 曲线异常时，先排查是分桶起点偏差还是pLTV排序精度问题，而非直接归因于出价档位设置错误

九、层面二：策略版本整体对照机制

9.1 缺口描述

现有机制只在"同一版策略内部"做10桶赛马，没有回答：这一整版策略相对上一版策略，整体是否更优？策略持续迭代（如5月→6月参数调整，或pLTV算法升级），每次迭代后无法证明新版本真的比旧版本好。这是当前验证体系的核心缺口。

9.2 为什么不能在媒体10桶上限内做版本对照

媒体侧的10桶上限是硬约束，无法在同一组10桶内同时编码"出价档位"和"策略版本"两个维度（即无法做到"版本A的10桶 + 版本B的10桶"同时存在）。因此版本对照必须跳出单次RTA实验的10桶框架，用独立的流量入口实现。

9.3 推荐方案：独立分流做整体AB对照

总流量
 ├── 主流量（80%~90%）→ 当前在用策略版本（如5月版）→ 10桶梯度按层面一方式运行
 └── 对照流量（10%~20%）→ 待验证新版本（如6月版）→ 完整复刻一套10桶梯度独立运行

对照逻辑：
    不逐桶比较（5月1号桶 vs 6月1号桶意义有限，两版pLTV排序逻辑本身可能已变化）
    而是比较"整体加权指标"：
        整体ROI        = Σ(各桶实际转化价值) / Σ(各桶出价成本)
        整体b浓度加权值 = Σ(各桶b浓度 × 该桶流量占比)
        整体出价准度偏差 = (整体实际转化价值 - 整体出价成本) / 整体出价成本

9.4 分流方式：双层独立哈希

Step 1：版本分流（新增一层，独立于桶内梯度分流）
    version_group = hash(unified_uid + version_salt) % 100
    若 version_group < 15  → 对照流量（待验证新版本）
    否则                   → 主流量（当前在用版本）

Step 2：各版本内部按层面一逻辑做10桶梯度分配
    bucket = hash(unified_uid + bucket_salt) % 10

关键：两次哈希使用不同的 salt，保证版本分流与桶内分流相互独立

9.5 版本对照的统计检验要点

事项说明
样本量估算版本对照效应量通常比单桶浓度差异更微弱，需基于历史数据估算所需样本量和观察周期，不能凭经验设定
混杂因素控制同一时间窗口并行跑两个版本，优于纯前后时序对比，因前者控制了季节性/媒体规则变化等时间维度混杂
观察周期至少覆盖一个完整业务周期（工作日+周末），避免短期波动误判，建议≥7天
pLTV升级场景分层口径本身已变化，5月/6月的"1号桶"可能代表不同客群；优先用整体指标对比，不可逐桶直接类比

9.6 补充手段：前后对比（仅作辅助）

若业务无法接受切出独立对照流量，可退而求其次用前后时间序列对比，但需明确：无法排除时间窗口内的媒体流量结构变化、季节性因素干扰，只能得出"方向性参考"，不建议作为版本是否推全的最终依据。

十、两层验证的衔接与决策流程

│   校验响应曲线单调性、边际递减、拐点位置                         │

                            ↓
┌────────────────────────────────────────────────────────────┐
│ 阶段3：分阶段放量推全                                          │
│   对照流量比例逐步提升：20% → 50% → 100%                      │
│   持续监控整体指标是否随流量增长衰减                             │
│   推全后保留小比例旧版本作为常驻哨兵                             │
└────────────────────────────────────────────────────────────┘
关键原则：先做层面二（版本整体是否更优），再做层面一（版本内部怎么调权）。如果新版本整体都不如旧版本，那么在新版本内部做再精细的桶内赛马也没有意义；反过来，桶内赛马调出的"性价比最优区间"，其参考价值也建立在"这一整版策略本身是有效的"这个前提之上。

十一、风险提示与待对齐事项

11.1 主要风险

风险项描述应对
对照流量探索成本 独立分流做版本对照，对照流量在观察期内使用"待验证"策略，可能短期效果不如主流量 业务方需接受一定探索成本；流量占比越小影响越小，但周期越长
样本量 vs 观察周期权衡 对照流量占比与所需观察周期是一对权衡，不能凭经验设定 基于历史方差提前做样本量估算，再确定流量占比和观察窗口

11.2 待与算法/数据团队对齐事项

  1. 版本对照所需的独立流量切分能力——媒体接口/RTA网关侧是否已支持在10桶之外再加一层版本分流标识，还是需要工程改造。
  2. 整体ROI、整体出价准度偏差率等版本对照指标的计算口径，需与现有桶内b浓度指标统一标准，避免两层验证用不可比的指标定义。
  3. 版本对照所需样本量/观察周期估算，需基于历史数据先做一轮基线方差测算。
  4. pLTV算法升级后，是否需要在升级窗口期保留双轨打分（新旧算法并行），以便版本对照期间有可比的分层基准。
  5. ID Mapping服务的实时性与覆盖率 SLA 确认，确保主实验人群的userid匹配率满足样本量需求。
