整体架构：Tanx 统一封装层

架构设计

Tanx 作为统一 RTA 封装层，屏蔽各上游媒体的接口差异。天渠只需对接 Tanx 一套协议，即可覆盖全部媒体流量，并享有统一曝光监测与转化回传能力。

🎯 Tanx 统一能力

  • 统一 RTA 请求协议（天渠无需适配各媒体）
  • 统一曝光监测（各媒体曝光 URL 托管）
  • 统一转化回传事件（激活 / 注册 / 付费）
  • 屏蔽媒体接口差异，降低接入成本

📦 下发字段（Tanx → 天渠）

  • device_id：设备 ID（OAID / IDFA / IMEI）
  • rtaid：分桶标识（腾讯/头条站内直接对应分桶）
  • platform：投放平台标识
  • site_id：版位 ID（用于 pLTV 三元组查询）
ℹ️ 分桶映射规则：腾讯广告和头条站内 RTA，每个 rtaid 直接对应一个分桶 ID，天渠无需二次取模；其他媒体按 uid 或 device_id Hash 取模分桶。

对接流程：三阶段七步骤

协作流程
阶段一：技术联调
→
阶段二：媒体配置
→
阶段三：放量验证
  • 1
    阶段一 · 技术联调

    天渠开发 Tanx RTA 接口

    实现符合 Tanx repeat 协议的 RTA 服务：接收 device_id + rtaid → ID Mapping → pLTV 查询 → 策略匹配 → 返回 is_bid + price_factor。响应超时要求 P99 ≤ 50ms，需支持 HTTPS POST + token 鉴权。

  • 2
    阶段一 · 技术联调

    Tanx 支持天渠联调（数据 Mock）

    Tanx 提供各媒体类型 Mock 请求（腾讯/头条站内/穿山甲/百度/快手），覆盖 OAID / IDFA / IMEI 三类设备 ID，验证超时、无效 ID、rtaid 不存在等异常场景。联调通过标准：各路请求正常解析，P99 响应 ≤ 50ms，repeat 协议 100% 合规。

  • 3
    阶段二 · 媒体配置

    Tanx 沟通确认媒体开户，分配策略 ID / 账户 ID

    各媒体分配对应 rtaid（腾讯/头条站内每个 rtaid = 一个分桶）；头条需单独分配站内 RTA 和穿山甲 RTA 两套独立 rtaid 集合；获取各媒体账户 ID / 策略 ID 清单并存档。

  • 4
    阶段二 · 媒体配置

    天渠侧配置策略 ID / 账户绑定的人群包信息

    在天渠配置中心完成：rtaid → 媒体类型 → 分桶 → 实验组 → 策略（pLTV Tier 出价系数表）的完整绑定；头条站内 & 穿山甲独立人群包，实验互斥配置，禁止共用。

  • 5
    阶段二 · 媒体配置

    天渠在媒体账户配置 Tanx RTA 接口 & 投放计划

    在各媒体广告后台填入 Tanx 提供的 RTA 接口地址（天渠服务端点），绑定 rtaid 至对应广告计划 / 广告组；头条两套 rtaid 分别绑定各自独立计划，确保数据不串流。

  • 6
    阶段三 · 放量验证

    天渠在媒体账户充值放量，测试链路

    灰度策略：1%~5% 小流量验证链路连通 → 10%~20% 核查分桶均匀性 & ID Mapping 覆盖率 → 50%+ 正式放量，同步上线监控面板和超时告警（超时率 > 5% 自动降级）。

  • 7
    阶段三 · 放量验证

    投放效果验证（业务评估指标）

    依托 Tanx 统一转化回传，评估四项核心指标：展（曝光/CPM）· 点（CTR/CPC）· 消（消耗/ROI）· 转化（激活率/付费率/CPA）；通过 rtaid + exp_id 归因至实验组，完成策略赛马显著性检验（p < 0.05）。

实时竞价数据流

核心链路

① 广点通发送 RTA 请求 → Tanx

上游媒体（腾讯/头条/百度/快手）在竞价时向 Tanx 发起 RTA 请求

② Tanx 封装转发 → 天渠 RTA 服务

Tanx 将媒体请求统一封装，携带 device_id、platform、site_id、rtaid 等字段，通过 HTTPS POST 下发至天渠

③ 天渠 RTA 服务执行内部处理逻辑（详见下方）

步骤 a ~ j 共十个处理环节，完成从设备 ID 到出价系数的全流程

a. ID Mapping

device_id（OAID/IDFA/IMEI）→ ID Mapping 服务 → 内部 uid；未命中则进入 deviceid 分支

b. 解析请求上下文

提取 platform（投放平台）和 site_id（版位），作为 pLTV 三元组查询的关键维度

c. 查询用户标签（pLTV 三元组 + 其他标签）

详见第四节 pLTV 三元组规范；其他标签：活跃度、购买力、新客标签、B会员标签（全局维度）

d. 匹配 RTAID

根据 platform 字段确定 RTAID：gdt→GDT、toutiao_inapp→TT站内、pangle→穿山甲（独立体系）

e. Zeus 实验分流 → 确定分桶

uid（或 device_id）经 Zeus 分流 Hash 取模，确定用户所在分桶号（腾讯/头条站内：rtaid 直接对应分桶）

f. 读取策略配置（Redis 缓存）

Key: strategy:{rtaid}:{bucket_id}，读取 bid_mode（人工/算法）、premium_rules、fallback_factor

g. 匹配溢价条件（标签 AND 组合）

逐条遍历 premium_rules，首条命中即停止；pltv_rank 的语义严格绑定当前 platform × site_id（如 pltv_rank > 0.01 在优量汇 site_id=15 = TOP 1%）

h. 计算出价（人工 / 算法双轨）

人工出价：直接返回配置系数；算法出价：传入特征（含 platform + site_id + 分媒体 pLTV）调用算法服务；超时走 fallback_factor 兜底

i. 返回结果给 Tanx（repeat 协议）

{ bid_id, is_bid, price_factor, rtaid, exp_id, tier, media }，全程 P99 ≤ 50ms

j. 异步写日志到 Kafka

含 request_id、uid、rtaid、bucket_id、strategy_id、platform、site_id、出价结果、命中规则、pLTV 值及来源标识（site_level / global_fallback / zero）

④ Tanx 接收出价 → 参与上游媒体竞价

is_bid=true 按 price_factor 出价；is_bid=false 不参与竞价

⑤ 日志数据流：Kafka → Flink → ODPS → 实验报告

Flink 实时聚合按 rtaid × bucket_id 的参竞率、关联 Tanx 统一回传的曝光/点击/转化，写入 ODPS 生成实验报告

// Tanx 下发请求示例 { "bid_id": "uuid-xxx", "rtaid": "tanx_rta_gdt_001", // 腾讯: rtaid = 分桶ID "device": { "device_id": "oaid_or_idfa", "id_type": "OAID" }, "ad_slot": { "platform": "gdt", // 投放平台 "site_id": "15" // 版位ID（优量汇=15） } } // 天渠 repeat 响应示例 { "bid_id": "uuid-xxx", "is_bid": true, "price_factor": 1.3, "ext": { "rtaid": "tanx_rta_gdt_001", "exp_id": "exp_gdt_001", "bucket_id": 3, "tier": 2 // 命中 pLTV Tier 2 } }

pLTV 三元组查询规范

核心规范
⚠️ 关键约束：pLTV 类标签必须传入 uid + platform + site_id 三元组精准查询，禁止跨媒体点位复用 pltv_rank 值。
// pLTV 查询 KEY 格式 KEY: pltv:{version}:{uid}:{platform}:{site_id} // 示例：pltv:v3:user_001:gdt:15
// 三级降级策略 Level 1 → 分点位数据 pltv:{version}:{uid}:{platform}:{site_id} ✅ 优先读取 Level 2 → 全局 pLTV pltv:{version}:{uid}:global ⚠️ 无分点位时降级 Level 3 → pLTV = 0 ❌ 全局也无时兜底
// pltv_rank 语义示例 pltv_rank > 0.01 in 优量汇(site_id=15) → 该用户在优量汇点位 pLTV TOP 1% pltv_rank > 0.01 in 穿山甲(site_id=xx) → 该用户在穿山甲点位 pLTV TOP 1%
字段 含义 降级说明 日志来源标识
pLTV_V3 V3 模型分点位 pLTV 绝对值 无则读全局，全局无则为 0 site_level
pLTV_V4 V4 模型分点位 pLTV 绝对值 无则读全局，全局无则为 0 site_level
pltv_rank 该媒体×点位下的百分位排序 全局降级时使用全局排序值 global_fallback
其他标签 活跃度/购买力/新客/B会员 全局维度，无需区分点位 —

头条双轨设计：站内 RTA vs 穿山甲 RTA

特殊处理

🔶 头条站内 RTA（toutiao_inapp）

  • 独立 rtaid 集合，每个 rtaid 直接对应一个分桶
  • 独立实验组 / 对照组配置
  • 独立人群包，不与穿山甲共用
  • 独立广告计划，不与穿山甲混投
  • 日志中 platform = "toutiao_inapp"

🔷 穿山甲 RTA（pangle）

  • 独立 rtaid 集合，独立于站内体系
  • 独立实验组 / 对照组配置
  • 独立人群包，不与站内共用
  • 独立广告计划，不与站内混投
  • 日志中 platform = "pangle"
⚠️ 严格隔离要求：头条站内与穿山甲的 rtaid、分桶、策略、人群包、广告计划、实验报告全部独立维护，禁止任何维度混用，否则将导致实验归因错误。

  • repeat 协议格式 100% 合规
  • 异常场景覆盖（超时/无效ID/rtaid不存在）
  • pLTV 三级降级逻辑验证通过

媒体配置 & 放量阶段

  • 各媒体 rtaid 已录入配置中心，分桶映射正确
  • 头条站内 & 穿山甲 rtaid 独立建档，实验互斥
  • 媒体账户 RTA 接口地址配置正确
  • 1% HoldOut 空白组已隔离
  • 分桶均匀性验证（χ² 检验 p > 0.05）

  • 超时告警规则上线（超时率 > 5% 自动降级）

关键设计原则

工程规范

🎯 pLTV 三元组精准性

pLTV 查询必须传入 uid + platform + site_id 三元组，精准到媒体×点位维度，避免跨点位数据污染影响溢价决策准确性。

📊 pltv_rank 语义绑定

pltv_rank 的百分位排序语义严格对应当前请求的 platform × site_id，不得跨媒体或跨点位复用同一 rank 值做策略判断。

🔒 头条双轨全隔离

头条站内 RTA 与穿山甲 RTA 在 rtaid、分桶、策略、人群包、广告计划、实验报告六个维度全部独立，禁止任何混用。

🛡️ 算法出价超时兜底

算法服务超时不允许无出价返回，必须在规定时间内切换至 fallback_factor 兜底系数，确保参竞率稳定。

📝 pLTV 来源必须日志

每次出价日志必须记录 pLTV 数据来源（site_level / global_fallback / zero），用于数据质量监控和排查分点位覆盖率问题。

🔗 Tanx 统一回传依赖

曝光/点击/转化事件统一依赖 Tanx 回传，不单独对接各媒体事件接口，通过 exp_id + rtaid 精准归因至实验组。
