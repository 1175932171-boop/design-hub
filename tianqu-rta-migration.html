问题一：成本高企，能力重叠，收费机制不合理

奥格收费机制
按请求量 × 人群包数量计费。同一请求命中多个人群包则多次计费。LTV策略上线后人群包已达 400+，靠人力合并已无法管控；多策略共用多个人群包的情况普遍存在，费用持续膨胀。
巨浪收费机制
按 RTAID 参竞规模计费，同一请求参竞多个 RTAID 多次计费。通投任务收费最贵（即便仅排除黑名单人群）。RTAID 越多费用越高，商业化外投并入后 RTAID 数量将大幅增加。
能力重叠本质：天渠已集成 人群更新、设备转 user_id、人群包/标签/上下文判定、溢价策略命中判定 等全套能力，几乎等于巨浪 + 奥格。当前架构是巨浪接收媒体流量后，因为自身不支持溢价，不得不转发给天渠处理，形成三跳串联的资源浪费。

问题二：巨浪能力缺失，严重制约主营业务 A 的用增

问题点 具体表现 业务影响
实验隔离 业务 A 与业务 B 共用巨浪实验 ID，分桶策略互相干扰；且业务 A 无法接入宙斯实验平台（巨浪用业务 B 的实验 ID） 策略效果评估失准，用增迭代受阻，已提需求 25 年底巨浪仍未支持
IDM 匹配率 巨浪 IDM 仅识别近几个月业务 B 的 userid，覆盖率 60%；天渠整合多方能力可达 70%+ 拉新流量中 10%+ 无法进入实验，用增拉新数据不完整
自分桶覆盖 OPPO 应用市场、VIVO 信息流、VIVO 应用市场、荣耀应用市场、荣耀信息流等媒体支持自分桶，但巨浪不支持 这些媒体上业务 A 无法做 RTA 实验迭代，策略固化
数据安全 业务 B 主体与部分媒体有竞争关系，媒体数据不能下发巨浪；全量 RTA 请求数据难以获取 腾讯广告联盟点位无法利用；全网设备用户画像建设受阻

迁移目标与整体路径

核心目标：天渠直连媒体（直连 or Tanx 接入），去掉巨浪和奥格，将节省下来的成本用于采买数据、服务器及人力，提升用增算法模型效果和服务稳定性，支撑 1688 业务长期发展。

两阶段实施总览

第一阶段
天渠能力补齐 + 媒体逐步直连迁移
补足自分桶全媒体覆盖、QPS 扩容、直连/Tanx 接入能力，按媒体从小到大分批切量，逐步替代巨浪和奥格的流量承接角色。
第二阶段
RTA 实验架构重建，完善用增实验迭代能力
在天渠内实现业务 A / B 实验体系完全分离，业务 A 独立实验 ID 接入宙斯平台，自分桶全媒体覆盖，最终实现全媒体不走巨浪和奥格。

第一阶段：天渠能力补齐清单

对照图1→图2的功能差距，天渠需要新增或增强以下能力。以下按优先级排序：

P0
自分桶实验分流 — 全媒体覆盖 新增
天渠已有自分桶能力但覆盖不全。需对 OPPO 应用市场、VIVO 信息流/应用市场、荣耀应用市场/信息流等媒体完成自分桶适配，这些媒体在巨浪侧长期无法使用自分桶，迁移后可立即解锁增益。
P0
QPS 服务器扩容 新增
天渠当前流量略少于巨浪。承接巨浪全量后 QPS 将显著提升，需提前完成容量评估和服务器采买。建议设定压测目标：在峰值流量 ×1.5 安全系数下 RT 保持稳定，错误率低于 0.1%。
P0
直连媒体接入 / Tanx 接入 新增
目前天渠靠巨浪转发，无直连媒体能力。需完成两类接入：① 直连接入：适合小流量媒体（延迟更低，如 OPPO/VIVO/荣耀系列），② Tanx 接入：适合集团 APP 矩阵大流量（标准化协议，统一接口），集团内各 App 流量均走 Tanx 汇聚。
P1
媒体数据安全隔离通道 新增
业务 B 主体与腾讯广告等媒体存在竞争关系，导致联盟点位数据无法下发巨浪。天渠以业务 A 主体接入，需与腾讯广告等媒体确认数据合规性，开通数据下发通道，解锁全量 RTA 请求数据用于用户画像建设。
P1
增量评估实验分流 — 独立验证 已有
天渠已有增量评估分流能力，但需要核查是否存在对巨浪实验 ID 的隐性依赖。确保迁移后业务 A 的增量评估完全在天渠内部闭环，不依赖巨浪侧的分组信息。
P1
人群包/标签迁移 — 奥格 400+ 人群包的天渠化 新增
奥格上 400+ 人群包需要迁移到天渠标签体系。建议：① 梳理人群包依赖关系，合并使用相同标签逻辑的人群包（天渠支持标签逻辑合并，可减少冗余）；② 迁移期保留奥格双跑一段时间，A/B 验证命中率一致性。
成本节省估算：第一阶段每完成一个媒体的迁移，该媒体对应的巨浪 RTAID 参竞费用 + 奥格人群包请求费用即可同步停止计算。若第一批（OPPO/VIVO/荣耀）占巨浪流量约 15-20%，预计年化节省 1000万+。完成全量迁移后年化节省目标 7000万～1亿（视完成进度）。

媒体迁移优先级排序

迁移原则：从小到大，分批切量，每批保留流量回滚开关。切量节奏参考：每次切量比例不超过 10%，观察 RT / 错误率 / 命中率稳定后再进行下一批。

第二阶段：天渠 RTA 迭代能力概览

能力一：流量下发切换
从小媒体（OPPO/VIVO/荣耀）开始逐步直连或接入 Tanx，分批切量保留回滚开关，RT p99 和错误率稳定后再扩量。
能力二：自分桶实验能力
接入 Zeus 实验平台：自动生成分桶规则、自动化实验效果报表、自动化推全审批上线工作流，支持全媒体自分桶（含巨浪不支持的 OPPO/VIVO/荣耀等）。
能力三：设备统一标签 Tair 化
将 pLTV、BA 分层等常用标签写入 Tair，不过人群包，实时毫秒级查询，彻底替代奥格人群包模式，降低人群包请求计费，插件化策略更新。

三大任务闭环流设计

闭环一：RTAID 增改闭环

RTAID 增改闭环（与 Demo RTAID管理页对齐）
① 新建 RTAID→② 自动生成 Zeus 实验→③ 关联 RTA 实验名称→④ 配置实验流量%（各组分配：实验组1/2/对照/空白）→⑤ 点击"生成RTA策略"进入策略管理页配置各组策略→⑥ 生成新 RTAID 版本→⑦ 审批通过→⑧ 实时生效 ✓
关键设计：RTAID 在 Demo 的 RTAID 管理页中创建，关键字段：渠道（Tanx/巨量引擎/腾讯等）、RTAID 编号、RTAID 名称、关联 RTA 实验名称。RTAID 与实验的关联在此处配置，支持编辑和下线操作。

闭环二：实验增改闭环

实验增改闭环（与 Demo 实验管理页+策略管理页对齐）
① 新建RTA实验（实验管理页）→② 配置渠道/RTAID/实验名称/描述→③ 配置实验流量%及各组分配（生成RTA策略）→④ 策略管理页：配置策略类型/出价模式/条件标签→⑤ 实验运行（自动分流+数据采集）→⑥ 实验报告（自动化指标报表：B浓度/ROI_0/LTV_0/CAC/买家转化率）→⑦ 实验决策：推全 / 继续观察 / 下线实验→⑧ 推全：选 RTAID + 选策略 → 全量生效 → 实验自动下线→⑨ 实时生效 ✓
关键设计：新建实验弹窗中包含"生成RTA策略"按钮，点击后跳转到策略管理页为各组创建对应类型策略。实验运行后可点击"报告"查看完整实验报告，报告包含：①指标明细表（流量占比/花费/会员数/B会员数/B用户浓度/CAC/LTV_0/人均LTV_0/ROI_0/买家转化率，各组各行）②各组策略分桶数据②系统自动生成实验结论（推荐最优组）④实验决策三按钮：推全 / 继续观察 / 下线实验。点击"推全"可直接进入推全策略弹窗（选RTAID + 选策略 → 确认推全），推全后实验自动置为下线状态。

闭环三：实时参竞闭环

实时参竞闭环
① 媒体RTA请求（deviceid+RTAID）→② ID Mapping（device→userid）→③ Tair查询标签（pLTV/preB/BA分层）→④ 实验分桶判断→⑤ 人群分层匹配→⑥ 命中出价策略→⑦ 返回溢价结果/出价系数→⑧ 日志存储（请求级全量）✓
关键设计：Tair 标签查询替代人群包接口调用，平均 <5ms。全量请求日志存储支持全网设备用户画像建设，并为实验报表提供数据源。

RTA 策略约束与 10 桶分层设计

与 Demo 对齐说明：策略设计采用出价策略标签（条件标签）定义，而非固定桶号排序。一个 RTA 实验下可包含多个策略（实验组策略 / 对照策略 / 空白策略），每种策略类型独立定义条件标签与桶号映射，桶号由用户配置，媒体层 10 桶上限为硬性约束。

基本约束（媒体层硬性限制）

分桶上限：10 个桶是媒体层硬性约束。巨量、腾讯等主流渠道均为此硬限，不可绕过。天渠的实验分桶设计必须在这个工程限制内完成。各组（实验组、对照组、空白组）共享同一媒体的 10 个固定桶，每组的策略决定每个桶的出价逻辑。
策略类型三分法（与 Demo 对齐）：
实验策略：携带主动调权出价逻辑（按条件标签出价），用于验证新策略效果
对照策略：携带基线出价逻辑（如固定分桶策略 / 全局算法出价），作为参照
空白策略（空白组）：无策略，不出价，用于度量 RTA 整体增量效果

实验分组结构（与 Demo 对齐）

一个 RTA 实验可配置多个实验组 + 1 个对照组 + 1 个空白组，各组百分比由实验流量配置决定：

组别策略类型出价行为流量占比（示例）用途
实验组1实验策略 A按条件标签主动调权出价30%验证新出价策略的核心实验组
实验组2实验策略 B按条件标签主动调权出价（另一版本）30%同时验证第二种实验策略，支持多版本赛马
对照组对照策略固定基线出价（如媒体自定义 / 均匀出价）30%对照基准，计算实验增量
空白组无策略空白组不出价（完全不参竞）10%度量 RTA 全局增量 ROI，排除出价效果干扰

分桶均匀性保障：初始浓度一致性要求

核心原则：在 RTA 策略干预之前，每个桶中各 pLTV 分层的用户占比应当一致。即：若全量用户中 pLTV TOP 10% 用户占比 10%，那么每个桶中 TOP 10% 用户占比也应趋近 10%。分流方案必须保证这一点（见双层实验机制起点校验部分）。

③
确认推全：除空白组外，该 RTAID 关联的所有渠道请求全部切换为推全策略出价；实验状态自动设为下线
推全说明：推全为不可逆操作，推全后旧实验下线，策略全量生效。若需回滚，需重新创建实验并手动配置回退策略。

RTA 双层实验机制设计

核心设计原则：控制变量法要求"曝光前10桶用户价值分布保持均匀"是控制变量（消除混杂因素），"曝光后各桶竞得用户B浓度"是因变量（观察出价效果的结果），而"RTA出价系数"才是自变量（主动干预的变量）。两层实验分别回答不同问题，互不耦合。

两层实验总览

层级一：策略内部梯度校验
问题：这一版策略内，10档出价梯度中谁的性价比最优？
分流层：媒体固定 10 桶（hash(userid) % 10）
自变量：出价系数（桶1=4.0 … 桶10=0.4）
因变量：曝光后竞得用户的 B 浓度
前提：10桶初始 pLTV 分布需均匀（起点校验通过后才可开跑）
结论：B浓度最高的桶 = 出价最准的档位 → 优先放量
层级二：策略版本迭代校验（新增）
问题：新版策略（整体）是否优于上一版策略？
分流层：天渠平台自分桶（独立于媒体10桶），Step 5/6 切分
自变量：整版策略（V_n vs V_n+1）
因变量：整组 ROI_0 / LTV_0 / B浓度综合指标
约束：不在媒体10桶内做二维交叉（避免超出媒体桶上限）
结论：新版整组优于老版整组 → 推全新版

层级一：策略内部梯度校验（增强版：加起点校验）

为什么需要起点校验？若 10 个桶里用户构成本来就不一样（桶1天生分到消费能力强的老用户，桶10分到刚注册的新用户），即便不做任何出价干预，桶1的转化效果也大概率优于桶10。这时观察到的"B浓度差异"来自用户分布的混杂，而非出价干预的效果，赛马结论将完全失效。

起点校验方法

Step 1
分层用户预分配：实验启动前，对全量 userid 按 pLTV 模型分为 S/A/B/C 四个价值层级，确定各层用户占比（如 S:A:B:C = 5%:15%:30%:50%）
Step 2
分层内独立哈希分桶：在每个价值层级内部独立执行 hash(userid) % 10，而非对全量用户统一 hash。确保每个桶里 S/A/B/C 各层浓度严格一致（分层哈希 = Stratified Hashing）
Step 3
起点浓度校验：抽样验证 10 桶各层用户占比偏差 < 3%。若偏差超标，重新调整哈希函数，直至通过校验再正式开跑实验
Step 4
梯度赛马正式开跑：10 桶按 pLTV 排序绑定出价系数（桶1=4.0、桶2=3.5…桶10=0.4），开始曝光竞价
Step 5
B浓度观测与结论：观察各桶曝光后竞得用户的 B 浓度。桶 X 的 B 浓度高 → 出价 X 在这个价值区间买到了正确的人 → 可以放大该档位出价配额
分层哈希（Stratified Hashing）的必要性：对全量用户统一 hash(userid) % 10 在理论上期望均匀，但实际中用户 ID 分布规律性（如新用户 ID 连续递增）可能导致某些桶系统性聚集特定价值层用户。分层哈希通过在每个价值层内独立分桶，从机制上消除这种混杂，是起点均匀性的工程保障。

层级二：策略版本迭代校验（新增设计）

为什么现有机制无法回答版本迭代问题？现有机制将 10 个媒体桶全部用于梯度赛马（自变量是10档出价），媒体桶上限已用满。若再把"版本A vs 版本B"嵌入同一个 10 桶结构，就变成二维交叉设计，混杂了"价格差异"和"版本差异"两个自变量，实验结论无法归因。

解决方案：将版本比较提升到天渠平台层面（Step 5/6 的自分桶），完全独立于媒体10桶结构。

关键结论——为什么这样设计才是完备的：
单独有层级一：只能知道"这版策略内出价梯度是否合理"，但不能告诉你新版本是否优于历史。
单独有层级二：可以验证版本整体优劣，但如果策略内部出价梯度不合理，即使新版本胜出也无法知道"应该更大力投哪个档位"。
两层同时运行：既验证版本方向是否正确（层级二），也找到版本内最优出价点（层级一），构成完备的迭代闭环。

RTA 分流机制设计

核心问题：用 userid 分流还是 deviceid 分流？

场景 A：有 deviceid 且可映射到 userid
流量：已登录用户（天渠 IDM 覆盖的 70%+ 请求）
同一用户在头条/腾讯/百度/快手等媒体均有曝光，若用 deviceid 分流会导致同一用户在不同媒体分入不同桶，产生交叉污染。
结论：以 userid 哈希分桶，保证同一用户跨媒体全程在同一桶。
场景 B：有 deviceid 但无法映射到 userid
流量：未登录/新设备用户（IDM 无法识别，约30%）
一个 userid 可能对应多个 deviceid（换机、多设备），用 deviceid 分流同一用户的多个设备可能落入不同桶，破坏桶间用户均匀性。
结论：deviceid 哈希分桶（独立命名空间），应用 Default 策略，不进主实验分析。
场景 C：无法获取 deviceid
流量：特殊设备（隐私模式、iOS 新设备未授权等）
无法建立任何稳定用户标识，无法分配到确定桶位。
结论：走全局 Default 策略（历史最优策略兜底），不计入任何实验组。

实验架构：业务 A / B 独立实验命名空间

核心问题：业务 A 与业务 B 实验体系完全分离

当前痛点：主营业务 A 和业务 B 共用巨浪的业务 B 实验 ID，导致：① 任意 RTAID 的策略变更对两个业务均有影响；② 业务 A 无法接入宙斯实验平台（实验 ID 归属于业务 B）；③ 巨浪实验 ID 申请分离需求提出已久，迟迟未被支持。

目标架构：天渠内实验命名空间独立

主营业务 A — 独立实验层
独立实验 ID：业务 A 专属，可直接接入宙斯实验平台进行效果评估
自分桶全媒体覆盖：含 OPPO/VIVO/荣耀等当前巨浪不支持的媒体，实验迭代无死角
IDM 匹配率 70%+：天渠整合多方 IDM 能力，较巨浪提升 10%+，拉新流量覆盖更完整
全量 RTA 请求数据可获取：不受巨浪数据隔离限制，支持全网设备用户画像建设
独立增量评估对照组：业务 A 独立设置增量实验，不受业务 B 策略干扰
业务 B — 独立实验层
独立实验 ID：业务 B 专属，历史实验逻辑不受迁移影响
桶位隔离：与业务 A 完全独立的分桶空间，策略变更不互相干扰
现有分桶逻辑保持稳定：业务 B 以拉活为主，IDM 策略维持现状，逐步优化
实验历史数据查询保留：业务 B 原实验 ID 数据作为历史基线，支持回溯查询

QPS 突增 / 服务稳定性 高风险
描述
新媒体切入后天渠承载 QPS 骤升，若扩容不足可能引发超时/错误率上升，影响参竞结果

控制迁移节奏，3-4 个月完成第一批切换（此时可停止对应媒体的巨浪/奥格计费），按实际节省 ROI 动态调整迁移速度
回滚
每批次迁移均可独立回滚，不强制全量切换；如 ROI 不及预期可暂停某批次迁移
腾讯广告数据安全合规 中风险
描述
业务 B 主体与腾讯广告存在竞争关系，联盟点位数据合规性确认可能周期较长，影响第三批迁移
缓解
在第一批迁移启动时同步开展法务/商务对接，争取在第二批完成前拿到数据合规授权
回滚
该批次迁移失败不影响整体降本目标（第一批+第二批已覆盖主要流量）；该部分流量维持巨浪现状
直连媒体协议适配 低风险
描述
不同媒体的 RTA 直连协议可能存在差异，联调工作量不确定
缓解
优先选择 Tanx 标准化接入大媒体；直连小媒体优先选择已有合作经验的渠道（OPPO/VIVO 等已有参考）
回滚
直连联调期间流量不切换，完全联调测试通过后再切量，影响范围可控

整体回滚策略

分层回滚机制：
L1 媒体级回滚：单媒体切量异常 → 该媒体流量切回巨浪，其他媒体不受影响（分钟级）
L2 批次级回滚：同批次多媒体异常 → 整批次回滚，进入问题排查阶段（小时级）
L3 全量回滚：天渠侧系统性故障 → 全量流量切回巨浪，天渠进入修复模式（已有巨浪保底）

关键前提：在整个迁移期间，巨浪作为保底链路保持可用状态。直到第三批迁移验证稳定后，再正式启动巨浪下线流程。
